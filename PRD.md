### **2. 개요**

#### **2.1. 문제 정의**
콘텐츠 크리에이터 및 마케터는 최신 정보를 바탕으로 블로그 포스트를 작성하기 위해 자료 수집, 초안 작성, 이미지 제작, 기술적인 게시 과정(HTML 변환, 업로드 등)에 많은 시간과 노력을 투입하고 있습니다. 이 과정은 반복적이며, 특히 속도가 중요한 기술 트렌드나 뉴스 관련 콘텐츠 제작에 병목 현상을 유발합니다.

#### **2.2. 프로젝트 목표**
지정된 키워드를 기반으로 관련 최신 정보를 자동으로 수집하고, AI를 활용해 이미지와 텍스트 초안을 생성한 뒤, 최종적으로 Quarto를 통해 GitHub Pages에 블로그 포스트를 게시하는 전 과정을 자동화하는 시스템을 구축합니다. 이를 통해 사용자는 콘텐츠의 핵심 아이디어와 최종 검수 과정에만 집중할 수 있도록 합니다.

#### **2.3. 기대 효과**
- **콘텐츠 제작 시간 단축:** 아이디어 구상부터 게시까지 걸리는 시간을 획기적으로 줄여 콘텐츠 생산 속도를 극대화합니다.
- **반복 작업 최소화:** 자료 조사, 이미지 검색/제작, 마크다운 변환 등의 수동 작업을 제거하여 생산성을 향상시킵니다.
- **콘텐츠의 신속성 확보:** 최신 뉴스나 트렌드에 대한 반응형 콘텐츠를 그 어느 때보다 빠르게 제작하고 배포할 수 있습니다.

### **3. 시스템 요구사항**

#### **3.1. 시스템 아키텍처 (개념)**
![System Architecture Diagram](https://example.com/system_flow.png)
*A conceptual diagram showing the flow from keyword input to a published blog post.*

사용자가 키워드를 입력하면, 시스템은 ①정보 수집 → ②초안 작성 → ③게시의 3단계 파이프라인을 순차적으로 실행합니다. 각 단계의 결과물은 다음 단계의 입력으로 사용됩니다.

#### **3.2. 기능 요구사항**

**FR-1: 정보 수집 (Information Gathering)**
- **FR-1.1 (구글 검색):** 시스템은 입력된 키워드를 바탕으로 구글 검색을 실행하여 최신성과 관련성이 높은 상위 10개의 웹 페이지(뉴스 기사, 블로그 포스트 등)를 식별해야 한다.
- **FR-1.2 (콘텐츠 추출):** 식별된 웹 페이지의 본문 텍스트를 추출하고 요약한다.
- **FR-1.3 (결과 저장):** 수집된 원본 링크 목록과 요약 내용을 `search.md` 파일로 저장해야 한다. 이 파일은 2단계의 입력 자료로 활용된다.
- **FR-1.4 (이미지 생성):** 입력된 키워드를 의미적으로 해석하여 Gemini 이미지 생성 API에 전달할 적절한 프롬프트(Prompt)를 생성해야 한다.
- **FR-1.5 (이미지 저장):** API를 통해 생성된 이미지를 프로젝트 내 지정된 경로에 `image.jpg` 파일로 저장해야 한다.

**FR-2: 초안 작성 (Draft Generation)**
- **FR-2.1 (콘텐츠 합성):** `search.md`의 요약 내용을 바탕으로, 논리적인 흐름을 가진 블로그 포스트 초안을 생성해야 한다. 초안은 서론, 본론, 결론의 구조를 포함해야 한다.
- **FR-2.2 (마크다운 형식):** 생성된 초안은 제목, 소제목, 글머리 기호 등을 포함한 마크다운(Markdown) 형식이어야 한다.
- **FR-2.3 (이미지 삽입):** FR-1.5 단계에서 생성된 `image.jpg`를 블로그 포스트의 대표 이미지 또는 본문 내 적절한 위치에 삽입하는 마크다운 구문을 포함해야 한다.
- **FR-2.4 (초안 저장):** 완성된 초안을 `draft.md` 파일로 저장해야 한다.

**FR-3: 빌드 및 게시 (Build & Publishing)**
- **FR-3.1 (HTML 변환):** 시스템은 Quarto CLI를 호출하여 `draft.md` 파일을 HTML로 렌더링해야 한다.
- **FR-3.2 (결과물 출력):** 렌더링된 HTML 파일은 Quarto 프로젝트 설정에 따라 지정된 출력 디렉토리(예: `_site/` 또는 `docs/`)에 `index.html`로 생성되어야 한다.
- **FR-3.3 (자동 커밋 및 푸시):** 시스템은 렌더링된 결과물이 포함된 디렉토리의 변경사항을 자동으로 Git에 커밋(commit)하고, 설정된 GitHub Repository의 `gh-pages` 브랜치(또는 지정된 브랜치)로 푸시(push)하여 GitHub Pages에 배포를 완료해야 한다.

#### **3.3. 비기능 요구사항**
- **NFR-1 (오류 처리):** 정보 수집 실패, API 호출 오류, 파일 I/O 오류 등 각 단계에서 발생할 수 있는 예외 상황을 처리하고, 로그를 기록하여 사용자가 문제를 인지하고 디버깅할 수 있도록 해야 한다.
- **NFR-2 (설정 관리):** 구글 API 키, Gemini API 키, GitHub 레포지토리 주소, Quarto 프로젝트 경로 등 주요 설정 값들은 별도의 설정 파일(예: `config.yaml`)을 통해 관리되어야 한다.
- **NFR-3 (확장성):** 향후 구글 검색 외 다른 정보 출처(예: 특정 RSS 피드)나 다른 게시 플랫폼(예: Velog, Medium API)을 쉽게 추가할 수 있는 모듈식 구조로 설계되어야 한다.

### **4. 릴리즈 계획 (MVP)**

- **Phase 1 (핵심 파이프라인 구축):**
  - 키워드 입력부터 구글 검색, `draft.md` 초안 생성까지의 핵심 로직을 구현.
  - 이미지 생성 및 게시 과정은 수동으로 진행.
- **Phase 2 (이미지 및 게시 자동화):**
  - Gemini 이미지 생성 API를 연동하여 `image.jpg` 자동 생성 기능 추가.
  - Quarto 렌더링 및 GitHub Pages 자동 게시 기능 구현.
- **Phase 3 (고도화):**
  - AI 초안 생성 로직 개선 (글의 논리성, 가독성 향상).
  - 다중 키워드 지원, 예약 발행 등 사용자 편의 기능 추가.
  - 상세한 오류 처리 및 로깅 기능 강화.
- **Phase 4 (확장):**
  - 플러그인 아키텍처를 도입하여 정보 출처 및 게시 플랫폼 확장 기능 제공.
  - GUI(웹 인터페이스) 버전 개발 검토.
